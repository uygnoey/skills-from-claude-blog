## 이 글이 뭔가요
CodeRabbit이 Claude를 활용해 코딩 요청과 코드 생성 사이에 ‘계획(플래닝) 오케스트레이션 레이어’를 두고, 코드 생성 전에 팀이 검토할 수 있는 구조화된 계획/PRD를 만드는 방식을 설명합니다. 목표는 “컴파일되고 테스트는 통과하지만 의도와 다른 코드”를 줄이는 것입니다.

## 언제 유용한가요
AI가 만든 코드가 겉보기에는 정상이지만 실제 의도를 놓치는 경우가 잦거나, 구현 전에 요구사항과 가정이 명시적으로 정리되지 않아 방향이 흔들릴 때 유용합니다.

## 핵심 포인트
- 구현 전에 구조화된 플래닝 단계를 넣어 가정(assumptions)을 드러내고 이해관계자 정렬을 돕습니다.
- 계획 문서를 ‘공유 아티팩트’이자 코드 생성 전 품질 게이트로 취급합니다.
- 작업 복잡도에 맞춰 모델 티어를 배치합니다(오케스트레이션은 Opus, 구조화된 플래닝 단계는 Sonnet, 컨텍스트 요약 등 좁은 작업은 Haiku).
- 계획 품질을 평가하는 평가 하네스(harness)를 구축합니다(LLM judge 기반 점수 + 생성된 코드의 동작 여부/불필요한 범위 증가/완료까지 토큰 등).

## 번들 리소스
- 결과/가정/엣지 케이스/검증 기준을 추출하기 위한 재사용 가능한 ‘플래닝 레이어’ 프롬프트 템플릿
- 구현 전에 팀이 검토할 수 있는 “협업 PRD” 아웃라인 예시
