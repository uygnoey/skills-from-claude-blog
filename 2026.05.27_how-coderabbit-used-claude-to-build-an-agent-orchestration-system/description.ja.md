## この記事について
CodeRabbitがClaudeを使って、コーディング要求とコード生成の間に「計画（プランニング）オーケストレーション層」を挟み、実装前にチームがレビューできる構造化された計画/PRDを作る方法を紹介します。狙いは「コンパイルとテストは通るが意図と違うコード」を減らすことです。

## どんなときに役立つか
AI生成コードが動きそうに見えても意図を外すことが多い場合や、実装前に要件と前提（assumptions）が明文化されず方向性がぶれる場合に役立ちます。

## 主なポイント
- 実装前に構造化されたプランニング工程を入れて前提を明確化し、関係者の認識合わせを行う。
- 計画を共有アーティファクト兼、コード生成前の品質ゲートとして扱う。
- タスク複雑度に応じてモデル階層を割り当てる（オーケストレーションはOpus、構造化プランニングはSonnet、コンテキスト蒸留など狭い作業はHaiku）。
- 計画品質を評価する評価ハーネスを作る（LLM judgeによるスコア + 生成コードの結果指標：余計なスコープや完了までのトークンなど）。

## 同梱リソース
- 成果・前提・エッジケース・検証観点を抽出するための再利用可能な「プランニング層」プロンプトテンプレート
- 実装前にレビューするための「協働PRD」アウトライン例
