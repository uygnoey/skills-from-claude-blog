# 가이드: Claude 컴퓨터/브라우저 사용 모범 사례

이 가이드는 Claude로 컴퓨터/브라우저 사용 에이전트를 만들 때의 권장 기본값과 디버깅 요령을 요약합니다.

## 1) 이미지 크기와 스케일링
- 스크린샷은 전송 전에 다운스케일합니다.
- 간단한 기본값은 1280×720이며, Opus 4.7은 1080p 시작을 권장합니다.
- 너무 낮은 해상도(< 약 960×540)는 피합니다.
- 제한 내에서 화질을 최대화하려면 종횡비를 유지하는 “max API fit” 크기를 계산합니다.

## 2) 좌표 정확성
스크린샷을 리사이즈했다면, 모델이 반환한 좌표를 네이티브 화면 좌표로 다시 스케일링한 뒤 클릭을 실행합니다.

## 3) 메시지 포맷
message content 배열에서 텍스트 지시문을 이미지 블록보다 앞에 둡니다.

## 4) 작은 타겟
- UI가 조밀하면 줌을 활성화합니다.
- 타겟이 너무 작다면 키보드 내비게이션을 우선합니다.
- 4K+ 화면에서는 Opus 4.7(더 큰 이미지 예산)을 쓰거나 DPI/스케일을 낮추고, 필요한 영역만 캡처하는 방식을 고려합니다.

## 5) Thinking effort (컴퓨터 사용)
- Opus 4.7: 기본 `high`; 처리량/비용 우선은 `low`; 매우 어려운 원샷 작업만 `max`를 고려합니다.
- Claude 4.6: 기본 `medium`; 처리량/비용 우선은 `low`; 단순 워크플로에서 지연시간이 최우선이면 thinking 비활성화를 고려합니다.

## 6) 프롬프트 인젝션 안전
- 웹 콘텐츠는 신뢰하지 않습니다.
- `computer_20251124`의 내장 분류기를 우선 사용합니다.
- 되돌릴 수 없는 작업은 사용자 확인(human-in-the-loop)을 요구합니다.
- 권한 범위를 줄이고, 상세 로그를 남깁니다.

## 7) 장기 실행 컨텍스트 관리
원문에서 제시한 실전 기본값 예시:
- 안정적인 프리픽스에 캐시 브레이크포인트 1개, 최근 tool 결과에 3개
- 스크린샷 롤링 버퍼 정리(예: keep_n=3, 약 25장 단위로 정리)
- 약 15만 입력 토큰 전후에서 커스텀 프롬프트로 서버 사이드 컴팩션 트리거
