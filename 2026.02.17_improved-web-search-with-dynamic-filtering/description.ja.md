## この記事について

この投稿は、Claude Platform の web search / web fetch ツールの新バージョンを紹介し、検索・取得結果をモデルのコンテキストウィンドウへ取り込む前に、コード実行で必要な部分だけを残す「動的フィルタリング」が可能になったことを説明します。

## どんなときに役立つか

Web 検索のワークフローで不要な HTML や長いページが多く混ざって精度が落ちたり、トークン使用量が増えたりする場合に役立ちます。特に技術ドキュメントの参照や、引用・出典の検証のように「関連箇所だけ」を抽出したいタスクに適しています。

## 主なポイント

- Web 検索は複数サイトの HTML を丸ごと取得して推論することが多く、トークン消費が大きい作業です。
- 動的フィルタリングでは、web search / web fetch ツールが結果を後処理するためのコードを書いて実行し、コンテキストに入れる前に関連部分だけを残します。
- BrowseComp では精度が Sonnet 4.6 は 33.3% → 46.6%、Opus 4.6 は 45.3% → 61.6% に向上しました。
- DeepsearchQA では F1 が Sonnet 4.6 は 52.6% → 59.4%、Opus 4.6 は 69.8% → 77.3% に向上しました。
- 2 つのベンチマーク平均で、性能が 11% 改善し、入力トークンは 24% 減少しました。
- Claude API で Sonnet 4.6 と Opus 4.6 を使って新しいバージョンのツールを利用する場合、動的フィルタリングはデフォルトで有効になります。
