## 이 글이 뭔가요

이 글은 에이전틱 코딩이 기본 작업 방식이 된 뒤 Claude Code 엔지니어링 팀이 프로세스와 조직 운영 방식을 어떻게 바꿨는지 설명합니다.

## 언제 유용한가요

팀이 에이전틱 코딩을 도입하면서 계획 수립, 맥락 공유, 코드 리뷰, 역할 경계를 품질·안전성을 유지한 채로 재설계해야 할 때 유용합니다.

## 핵심 포인트

- 장기 로드맵 중심에서 벗어나, PR 논의와 프로토타입에 더 의존하는 just-in-time 계획으로 전환합니다.
- “작성자에게 물어보기” 대신 “먼저 Claude에게 물어보기”로 맥락 수집 방식을 바꾸고, 해당 단계를 자동화할 수 있는지도 함께 검토합니다.
- 코드 리뷰는 유지하되, 사람은 신뢰 경계·보안 민감 코드·법적 리스크·제품 감각처럼 중요한 부분에 집중합니다.
- 역할 경계가 흐려지는 것을 전제로 운영합니다(PM/디자이너의 코딩 증가, 엔지니어의 크로스펑셔널 업무 증가).
- 몇 가지 비협상 원칙을 정하고 세부 운영은 pod 단위로 조정하게 하며, 온보딩 램프업 시간/PR 사이클 타임/Claude 보조 커밋을 추적합니다.

## 번들 리소스

- 원문에는 함께 제공되는 파일이나 코드 예시가 없으므로, 이 폴더는 글의 핵심 내용을 재사용 가능한 skill과 템플릿으로 구성합니다.
